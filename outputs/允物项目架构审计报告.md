## 3. 数据库结构

### 数据模型清单（12 个模型 + 5 个枚举）

```
用户认证：
  User                        (邮箱,密码,角色,权限JSON,重置令牌)

产品体系（4层）：
  Series → Works → Products → ProductSku
    + ProductCost             (1:1 SKU 成本)
    + WorksAssets             (1:1 作品富媒体)

材料资产体系：
  RawMaterial                 (编码,类型,库存,成本,供应商,规格)
    + PurchaseRecord          (采购记录,加权平均成本)
    + InventoryTransaction    (库存事务 IN/OUT/ADJUST)

BOM 关联：
  Bom                         (SKU ↔ Material 多对多,含用量/成本快照)

客户与订单：
  Customer                    (客户信息,渠道,标签)
  Order                       (订单,渠道,状态,金额JSON items)
```

### 枚举定义
- `LifecycleStatus`: DRAFT / DESIGNING / READY / ACTIVE / PAUSED / ARCHIVED  
- `MaterialType`: BEAD / METAL / CERAMIC / LEATHER / INCENSE / CORD / PACKAGING / OTHER  
- `TransactionType`: IN / OUT / ADJUST  
- `OrderChannel`: MINIPROGRAM / WEBSITE / MANUAL  
- `OrderStatus` + `PaymentStatus`：标准订单生命周期

### 分析

**✅ 优点：**
- 模型分层清晰：产品体系（4层）→ 材料体系 → BOM 关联 → 订单客户
- 关系明确：1:1（ProductCost↔SKU, WorksAssets↔Works），1:N（Series→Works→Products→SKU），N:M（BOM）
- 枚举驱动状态机：生命周期 + 订单流转均使用 Prisma Enum
- 加权平均成本：PurchaseRecord 中实现了加权平均算法

**⚠️ 注意项：**
- `Order.items` 字段使用 `String` 存储 JSON（无结构化关联到 SKU/Product 表），属于半结构化设计
- `User.permissions` 同样用 JSON String 存储，关联松耦合
- 无 Prisma Migration 历史（已确认：`prisma/migrations` 目录不存在），当前使用 `prisma db push`
- 存在两份备份文件：`prisma/yunwu.db.backup.*`（SQLite 历史残留）

**冗余模型检查：** 无明显冗余。12 个模型各司其职，无重复定义。

**业务混合检查：** 本项目是纯粹的 **ERP/品牌管理系统**，仅包含：
- 产品生命周期管理
- 材料库存与采购
- BOM 物料清单
- 客户订单管理
- 用户权限系统

**不包含：** 电商前台、推荐系统、内容 CMS、AI/算法模块。

## 8. 架构依赖风险分析

### A. 当前架构是否模块化？

**是，但模块化程度中等。**

- ✅ API 层按 domain 分层（materials / orders / products / ...）
- ✅ 业务逻辑已抽出 `src/modules/`（目前仅 raw-material）
- ✅ 权限系统独立于 `lib/permissions.ts`，纯函数设计
- ⚠️ `src/modules/` 目前仅含 1 个模块（raw-material），其余 domain 的业务逻辑散落在 API route.ts 中
- ⚠️ 订单 domain 无对应 `src/modules/` 模块

### B. 是否存在明显"耦合严重"的情况？

**是，存在以下耦合点：**

1. **API 路由直连 Prisma**：大部分 route.ts 中直接 `prisma.xxx.findMany()`，无 Service 层抽象。这使得切换 ORM 或数据库需要修改所有路由文件。

2. **订单 items JSON String**：Order → items 字段存储序列化 JSON，与 Product/SKU 表无外键关联，数据完整性依赖应用层。

3. **import/export 大文件**：导入导出逻辑集中在单一文件，switch-case 分支管理，新增功能需修改同一文件。

4. **硬编码数据库凭据**：`package.json` 的 `db:push:staging` 和 `db:push:prod` 脚本中包含完整的 Neon 连接字符串（含密码），这是一个安全隐患。

### C. 数据模型是否稳定？

**是，数据模型稳定。**

- 12 个模型的职责边界清晰
- 枚举驱动状态机减少无效状态
- 使用 `@map` 和 `@@map` 规范化表名/字段名
- 无循环依赖或过度嵌套关系
- **但**：无 Prisma Migration 历史（使用 `db push`），缺少版本演进记录

### D. 推荐系统是否可以独立拆分？

**不适用。** 本项目不存在推荐系统、评分系统或规则引擎代码。全项目为 ERP 数据管理，无需拆分。

## 9. 迁移难度评分

# 📊 迁移到 Codex 难度评分：**3 / 10**

### 评分依据

| 维度 | 评分 | 说明 |
|---|---|---|
| 代码规模 | ★☆☆ | 仅 ~9,936 行 TS/TSX，75 个文件 |
| 架构复杂度 | ★☆☆ | 单体 Next.js App Router，无微服务 |
| 外部依赖 | ★★☆ | 仅依赖 Neon PG + Vercel，无 MQ/AI/WS/Redis |
| 数据库迁移 | ★★☆ | PostgreSQL 标准，Prisma ORM 可直接复用 |
| 业务逻辑 | ★☆☆ | 纯 CRUD + 简单加权平均，无复杂算法 |
| 平台绑定 | ★☆☆ | 无 Workbuddy 或特定平台锁定 |
| 代码质量 | ★★☆ | 38 处 `any`，部分路由直连 Prisma（缺 Service 层） |

### 最大风险点

1. **数据库凭据泄漏（P0）**：`package.json` 中的 `db:push:*` 脚本包含完整 Neon 连接字符串 + 密码，Git 提交历史中已存在。
2. **Order.items JSON 字段**：半结构化存储，外键缺失，数据迁移时需要额外处理。
3. **无 Migration 历史**：`prisma db push` 方式导致无 schema 演进记录，Codex 迁移时需要重新建立 migration 基线。
4. **PrismaClient 多实例**：scripts/ 中独立 `new PrismaClient()`，迁移时需统一。

### 建议

> **推荐：直接迁移**（而非重建）

理由：
- 代码规模小（<10K 行），重写成本与迁移成本接近
- 架构简单（单体 Next.js），学习曲线平缓
- 无"高耦合不可拆分"的模块
- 外部依赖极简（PostgreSQL + 标准 npm 包），无供应商锁定
- 建议迁移顺序：
  1. 先建立 Prisma Migration 基线（`prisma migrate dev --create-only`）
  2. 移除 `package.json` 中的硬编码凭据（改用 `env()` 引用）
  3. 为每个 domain 补充 Service 层（对齐 `src/modules/raw-material/` 模式）
  4. 迁移 Order.items 为关联表（可选，视 Codex 是否需要）
  5. 统一 PrismaClient 实例化为 `lib/prisma.ts`
